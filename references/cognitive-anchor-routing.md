# 内容锚点与视觉路由

本文件用于把一篇文章从“文字段落”转成“应该画什么”。先做判断，再写 prompt；不要先套构图。

## 逐节配图判断表

处理文章、长文、公众号链接、项目复盘或教程时，先逐节判断：

| 字段 | 要求 |
|---|---|
| 小节/段落 | 保留原文结构，不凭空重排 |
| 内容信号 | 观点、步骤、冲突、案例、数据、情绪、总结 |
| 读者卡点 | 非专业读者可能不理解、记不住或误判的地方 |
| 配图判断 | `适合配图` / `不适合配图` / `适合卡片` / `适合封面` |
| 推荐路由 | `emotion-anchor`、`explanatory-diagram`、`technical-architecture`、`process-flow`、`comic-strip`、`knowledge-card-pack`、`knowledge-card-single`、`infographic-poster`、`platform-cover`、`xinghe-article`、`prompt-only` |
| 理由 | 为什么画，或为什么不画 |

双向原则：

- 不为了凑图而画：定义、背景铺垫、重复口号、已经足够清楚的短列表，可以不配图。
- 不错过难点：抽象概念、机制、关键步骤、前后对比、冲突转折、核心方法，应该主动给视觉方案。

## 深度提炼五问

生成视觉方案前，先回答：

1. 文体是什么：观点、教程、复盘、产品介绍、方法论、故事、发布包。
2. 真意是什么：这篇内容真正想让读者理解或采取什么行动。
3. 张力在哪里：理想与现实、误区与真相、混乱与秩序、人工与自动化、输入与输出。
4. 灵魂句是哪一句：最适合成为封面标题、卡片标题或画面短标注。
5. 必须出现什么：原文术语、数字、步骤、案例、风险边界和用户确认标题。

## 自主路由编排

读完整篇文章后，先由内容决定图型，不要让用户先替你选图型，也不要默认只输出一种正文图。路由编排分三层：

| 层级 | 说明 |
|---|---|
| `primary_route` | 当前文章或当前配图点最主要的视觉形态，只能有一个 |
| `secondary_routes` | 可以辅助主路由的视觉形态，可以为空，也可以有 1-2 个 |
| `recommended_outputs` | 建议输出几张图、每张图承担什么阅读任务 |

每次路由决策必须说明：

- 为什么这个主路由最适合当前内容。
- 是否需要辅助路由，以及辅助路由解决什么问题。
- 信息密度是 `low`、`medium` 还是 `high`。
- 读者先看哪里、再看哪里、最后记住什么。
- 这张图最容易失败在哪里：太密、太散、太像 PPT、情绪过火、人物遮挡结构、文字太多或关系不清。

## 路由选择

| 路由 | 什么时候用 | 不要误用 |
|---|---|---|
| `platform-cover` | 文章第一眼点击、公众号封面、小红书首图 | 不要把正文图硬拉成封面 |
| `emotion-anchor` | 情绪冲突强、误区真相强、理想现实反差强 | 不要只画表情包 |
| `explanatory-diagram` | 抽象概念、因果、步骤、轻结构、机制解释 | 不要做冷冰冰 PPT |
| `technical-architecture` | 系统组件、服务边界、数据流、模块依赖、部署关系、权限边界 | 不要为了人物牺牲节点关系 |
| `process-flow` | SOP、自动化流程、审批流、状态流、内容生产流程、任务流 | 不要把长流程硬塞进一张小卡 |
| `comic-strip` | 有因果、累积、转折、前后变化、节奏感 | 如果删一格不影响理解，就降级 |
| `knowledge-card-pack` | 长文需要轮播卡、收藏卡、文章总结卡 | 不要把全文硬塞进一张图 |
| `knowledge-card-single` | 单观点、单流程、单对比、单清单 | 不要把卡片当普通插画 |
| `infographic-poster` | 整篇流程、矩阵、地图或全局总览 | 默认最多一张 |
| `xinghe-article` | 常规正文锚点图，适合一图一概念 | 不要自动默认为 16:9 |
| `prompt-only` | 用户只要提示词，或无法上传人物基准图 | 不要假装已经生成 |

## 组合路由

有些文章不是单一图型可以讲清楚，允许组合路由，但要分清主次：

| 组合 | 什么时候用 | 输出方式 |
|---|---|---|
| `comic-strip + emotion-anchor` | 内容有误区、压力、转折、松一口气、从混乱到清楚等情绪变化 | 漫画是主路由，情绪是每格的驱动力；每格写清情绪阶段和动作变化 |
| `infographic-poster + knowledge-card-pack` | 文章信息量大，既需要全局地图，也需要分张阅读 | 信息图做总览，知识卡片承接细节；不要把全文塞进海报 |
| `knowledge-card-single + explanatory-diagram` | 多个相关知识点需要放在同一张卡里，并且彼此有关系 | 卡片是容器，解释图是内部关系骨架 |
| `process-flow + knowledge-card-pack` | 流程步骤多、检查点多、需要分段理解 | 流程拆成卡片组，每张卡讲一段流程或一个关键节点 |
| `technical-architecture + process-flow` | 既要讲系统由什么组成，又要讲执行怎么跑 | 拆成架构图和流程图，或主图讲架构、侧栏讲主路径 |
| `platform-cover + emotion-anchor` | 封面需要明确情绪张力或反差，帮助点击 | 封面仍以标题和核心判断为主，情绪只做承托 |

组合路由不是把几种图型硬拼在一张图里。若主路由已经能讲清楚，辅助路由只作为构图和情绪策略；若信息量过高，应拆成多张图。

## 自主选择规则

- 情绪强、矛盾强、读者需要先共鸣：优先 `emotion-anchor`，若有前后变化再考虑 `comic-strip + emotion-anchor`。
- 抽象概念、机制、因果和轻量步骤：优先 `explanatory-diagram`。
- 系统组件、数据流、权限、部署、模块边界：优先 `technical-architecture`。
- SOP、状态流、审批流、内容生产流、自动化执行链：优先 `process-flow`。
- 多个相关知识点需要收藏、复习、分张阅读：优先 `knowledge-card-pack`。
- 单个核心观点、对比、流程或清单需要沉淀：优先 `knowledge-card-single`。
- 全文是地图、矩阵、方法全貌或路线图：优先 `infographic-poster`，信息密度可以是 `high`，但必须分层分区。
- 第一眼点击或平台首图：优先 `platform-cover`，正文图不要硬改成封面。

技术架构图、流程图和高密度知识卡片要额外判断人物呈现等级：

- `full-character`：封面、情绪图、正文锚点图，人物承担主要动作。
- `small-character`：解释图、流程图、架构图，人物只做标注、指向、接线。
- `partial-character`：知识卡片、流程节点密集图，只露手、半身或侧影。
- `no-character`：结构精确或信息密度高，人物会干扰理解；输出为星禾风格结构图。

## 来源锁定

画面里的标题、术语、数字、步骤和关键标签必须来自：

- 用户明确给出的标题或正文。
- 原文中可核对的词语。
- 用户确认过的改写。

不能为了画面好看编造数字、承诺、案例、平台功能或商业效果。

## 参考项目原则

只学习方法，不复制角色、画风、图片和文字：

- 文字渲染铁律：文字少、准、来自原文；错字优先重生。
- 叙事统一：一篇内容内的视觉隐喻要属于同一个世界。
- 顶层哲学：方法论恒定，角色与场景是参数。
- 极简场景：需要解释结构时，可以让人小、物大、结构清楚。
