# 中文文字渲染规则

生图里的中文文字是高风险区。默认少写、写准、写短。

## 核心原则

先治理文字，再写 prompt。任何要进入画面的中文，都必须先形成 `text_inventory`，不要边写 prompt 边临时发挥。

每张图都要回答：

- 这张图必须出现哪些字？
- 哪些字来自原文、用户标题或用户确认？
- 哪些字只是辅助标签，是否可以删掉？
- 如果模型写错，是否能接受重生？如果不能，就减少文字或改成无文字结构图。

## 文字来源

允许写入画面的文字：

- 用户明确给出的封面标题。
- 原文里的关键术语、步骤名、数字、对象名。
- 用户确认过的改写标题或短标签。

不要写：

- 未核验数字。
- 未出现过的商业承诺。
- 二维码、联系方式、价格、购买引导。
- 色值、字体名、提示词参数或模型术语。
- 大段正文、完整代码、密集表格。

## text_inventory

写 prompt 前先整理文字清单：

```json
{
  "text_inventory": {
    "title": "用户确认标题",
    "must_keep_terms": ["Codex", "工作区系统", "Agent Loop"],
    "numbers": ["3 步", "7 个检查点"],
    "short_labels": ["输入", "编排", "执行", "交付"],
    "forbidden_or_avoid": ["保赚", "唯一", "永久有效"],
    "text_budget": "1 标题 + 5 个短标签 + 1 句底部提示"
  }
}
```

规则：

- `title` 必须逐字来自用户确认版本；没有确认时只给候选，不写入真实生图 prompt。
- `must_keep_terms` 保留大小写、空格和原文写法，例如 `Codex`、`API`、`Agent Loop`。
- `numbers` 必须来自原文或用户确认；没有来源的数字不进画面。
- `short_labels` 每个尽量 2-6 个字；超过 8 个字时优先改写成更短标签。
- `forbidden_or_avoid` 不进画面；如原文必须讨论风险，用中性词替代，例如“承诺风险”“需人工确认”。

## 字数密度

| 图形 | 建议文字 |
|---|---|
| 情绪锚点图 | 1 个短标题或 2-5 个短标签 |
| 解释图 | 3-7 个节点短词，每个 2-8 字 |
| 多格漫画 | 每格 0-1 句短对白或短标签 |
| 知识卡片 | 1 个主标题 + 3-5 个要点 + 可选底部总结 |
| 信息图海报 | 分区标题短，正文尽量用关键词 |
| 全景信息图 | 7-10 个信息区，18-28 个短标签；禁止段落化 |
| 技术架构图 / 流程图 | 节点名 2-8 字，边标签尽量 0-4 字 |
| 公众号封面 | 1-2 行短标题 |
| 小红书封面 | 2-4 行大字标题 |

如果文字超过限制，优先拆成多张卡或多张图。

## 标题规则

- 标题必须逐字来自用户确认版本。
- 需要英文时保留原文大小写，例如 Codex、AI、API、Workflow。
- 标题不要贴边，移动端保留安全边距。
- 只强调 1 个关键词，使用下划线、圈注或轻色块。

## Prompt 写法

在生图 prompt 中明确写：

```text
Chinese text must be limited to the exact approved text inventory below.
Do not invent new Chinese words, numbers, slogans, contact info, QR codes, UI text, random English, or extra labels.
If text is hard to render, reduce labels instead of adding new words.

Approved Chinese text inventory:
- Title: "..."
- Labels: "输入", "编排", "执行", "交付"
- Numbers: "3 步"
```

不要写“画面里有很多中文说明”“加一些提示语”“丰富一点文字”。这类模糊指令会让模型乱造词。

## 高风险文字

以下文字进入画面前必须删改或人工确认：

- 绝对化：唯一、第一、最强、永久、必胜、稳赚、零风险。
- 商业承诺：保收益、保证成交、自动赚钱、躺赚。
- 隐私和联系：手机号、微信号、二维码、邮箱、地址、真实客户名。
- 敏感凭据：API key、token、secret、permission code、完整 endpoint。
- 未核验事实：具体价格、排名、数据、发布日期、公司承诺。

如果文章本身要讨论这些风险，只能作为“风险标签”出现，例如“收益承诺风险”“隐私信息”“需人工审核”。

## 错字处理

- 错字少：可局部编辑或重生。
- 错字多：减少画面文字后重生。
- 不用代码在图片上涂改错字；这会破坏手绘质感和可信度。
- 标题错字、关键术语错字、数字错字：必须返工，不用评分抵消。
- 全景图和信息图错字多：先把段落拆成短标签，再重生。

## 代码、表格、UI

- 代码只能画成“代码卡片”或“命令纸条”的示意，不写长代码。
- 表格只能画成简化卡片或对比栏，不画密集真实表格。
- UI 只能抽象成低科技窗口或小白板，不生成真实 App 截图。

## 检查问题

交付前检查：

1. 画面文字是否都能在原文或用户标题中找到来源？
2. 手机端能否一眼读清？
3. 有没有错别字、乱码、随机英文或多余符号？
4. 是否出现了联系方式、二维码、价格或不该有的承诺？
5. 如果去掉一半文字，图是否更清楚？如果是，就删减。
6. 是否存在标题/术语/数字三类硬错误？如果有，直接返工。
7. 是否把 prompt 参数、色值、模型名、接口地址或密钥画进了图？如果有，直接返工。
