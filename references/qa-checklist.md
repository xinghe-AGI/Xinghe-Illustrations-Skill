# QA Checklist

封面任务不要用本清单作为主验收标准；公众号封面、小红书封面、文章头图和笔记首图读取 cover-qa-checklist.md。本清单用于正文配图、情绪图、解释图、技术架构图、流程图、多格漫画、知识卡片和信息图海报。

## 必过项

- 比例符合路由：知识卡片不能默认 3:4；小红书/微信竖图/手机轮播才优先 3:4，文章笔记/方法论总结/对比流程卡优先 4:3，横向流程/时间线/多模块关系/公众号正文横图卡优先 16:9，单概念可用 1:1。
- 知识卡片构图符合内容关系：路径、分层、决策、汇聚、对比、时间线或三列方案要选对骨架；不要所有卡都变成“标题 + 三点清单”。
- 背景是冷白或极浅蓝灰，允许轻量手绘分区、点阵、虚线和浅色模块承接信息。
- 人物呈现等级符合路由：正文图/情绪图/漫画通常有星禾；技术架构图、流程图、高密度知识卡片可以是小人物、局部人物或无人物。
- 默认只出现一个星禾人物；除非用户明确要求多人物叙事，不出现头像气泡、右下角 inset portrait 或重复星禾。
- 配图选点成立：这张图画在文章真正需要视觉锚点的位置，而不是平均凑图。
- 图型或路由判断成立：concept/process/comparison/data/scene/metaphor/handoff/review-loop/emotion-anchor/explanatory-diagram/technical-architecture/process-flow/comic-strip/knowledge-card/infographic-poster 与正文内容匹配。
- 自主路由成立：有清楚的 `primary_route`、必要时有 `secondary_routes`，信息密度、建议产物和路由风险解释得通；混合路由必须主次明确。
- 路由评分成立：`route_scores` 有分数、理由、风险和输出角色；最高分路由与最终 `primary_route` 一致，或说明了偏离原因。
- 文章类型成立：`article_type` 有可核对内容信号，不用默认类型套所有文章。
- 文字密度符合图型：封面极低、情绪图低、解释图中等、知识卡中高、信息图高密度但有分区；超出预算时拆图，不缩小字号硬塞。
- 星禾出现时承担核心创作、运营、判断、标注或引导动作，不只是装饰；结构类图无人物时，结构必须足够清晰。
- 星禾动作和当前图片内容绑定；不同主题不能复用同一个举瓶、挥手或站立姿势。
- 如果出现星禾人物、手部、半身或侧影，保留固定识别点，并贴近 `assets/examples/00-xinghe-ip-baseline.png`：黑色长发或微卷长发、轻薄空气刘海、温柔浅笑、白色宽松拉链连帽外套、深蓝水手领上衣、白色领结、深蓝百褶裙。
- 手部和握物关系自然：递卡、托卡、插线、按按钮、铺证据时，手指数量、手腕连接和物件遮挡不能穿帮。
- 身体比例和重心自然：头、肩、躯干、腿、脚的连接合理；蹲、跪、弯腰、伸手时不能拧成不可能的姿势。
- 不残留旧版本黑色小人主体；如果使用旧案例改图，旧主体必须被星禾替换。
- 没有复刻原参考案例构图，而是为当前文章生成新隐喻。
- 使用参考图时，人物形象来自 `00-xinghe-ip-baseline.png`，场景图只继承构图、动作、线条质感、留白和批注密度，没有照搬场景图旧物件、旧路径或旧布局。
- 画面亲切、清爽、易懂、有行动感。
- 简洁清爽，主体不超过画面约 60%。
- 一张图只讲一个核心结构、流程或判断。
- 中文标注少、短、能读。
- 橙色可以用于标题字、关键词、短标签、主路径、箭头或移动关系；检查重点是文字层级清楚，不是把橙色限制成强调线。不要把所有文字都做成橙色。
- 红色只用于断点、问题、提醒或结论。
- 蓝色只用于补充说明、AI/assistant/自动化状态、反馈或衣褶。

## 失败信号

出现以下情况，重生成或局部编辑：

- 左上角有“运营流程 / Workflow / 系统架构图 / 自动化 SOP / 研究框架 / 路线图”等标题。
- 配图位置不成立：这段内容并不需要图，或图没有比正文更清楚。
- 图型和内容错配：例如判断题画成流水线、证据题画成灵感瓶。
- 混合路由只是堆图型名：例如漫画没有情绪阶段、信息图没有分区阅读路径、知识卡片把无关系知识点硬塞在一起。
- 没有路由评分，或评分理由无法解释为什么选择当前图型。
- 知识卡片组没有叙事结构，只是一组散卡。
- 信息超出当前图型文字预算，却靠缩小字号硬塞。
- 星禾像儿童卡通、二次元海报或可爱头像。
- 出现多个星禾、头像气泡、右下角小头像、重复人物或把星禾画成头像贴纸。
- 星禾外形没有贴近 00-xinghe-ip-baseline.png，或服饰漂移成旧 hoodie/裤装/复杂 cosplay。
- 星禾外形像参考 IP，但动作和主题无关，只是在保持参考图的举瓶姿势。
- 多张图里星禾姿势高度重复，看起来像把同一个人物贴到不同图上。
- 手部穿帮：三只手、多指、断腕、手掌扭曲、手和卡片/工具粘连，或看不清她到底怎么拿住物件。
- 身体穿帮：上半身和腿不在同一重心、蹲跪姿势不可能、肩膀/手臂/腰胯连接错误。
- 画面像模板 PPT、课程课件或冷硬正式流程图，而不是冷白/浅蓝灰底蜡笔技术架构图、流程图、手绘解释、知识卡片或内容笔记。
- 技术架构图/流程图里完整人物过大，遮挡节点、箭头或文字。
- 技术架构图/流程图为了展示人物而牺牲结构清晰度。
- 元素太多、箭头太多、节点太多。
- 文字变成大段解释。
- 背景有纸纹、阴影、渐变、米色、噪点或大面积黄色。
- 解释图、流程图或知识卡片背景过空，信息像漂浮在白底上，没有分区、承载面或阅读路径。
- 知识卡片缺少主阅读路径：看不出从哪里开始、往哪里走、结论在哪。
- 真实 UI 截图或科技感界面。
- 中文错字严重或标注不可读。
- 画面太商业、太课件、太复杂。
- 星禾没有灵感、内容、运营或判断动作，只是在摆 pose。
- 和 `assets/examples/` 里的旧案例构图过于相似。
- 参考图把画面带偏：旧物件、旧箭头路径、旧中文标注或旧构图被明显复刻。

## 迭代方法

- 太普通：让星禾亲自执行举瓶、接线、排期、回流、分发、圈注或称量动作。
- 太复杂：删节点，只保留一个动作和 3-5 个短标注。
- 太可爱：强调 friendly, reliable, clear-minded, not childish, not mascot。
- 太 PPT：保留必要结构，但去掉模板感、冷硬边框和过多箭头，改成白纸上的手绘解释、技术架构图、流程图、卡片或内容笔记。
- 人物太抢：把人物降级为 `small-character`、`partial-character` 或 `no-character`，让节点、路径、层级先读清楚。
- 太像旧案例：保留核心意思，换掉主物件和星禾动作。
- 太冷：加入灵感瓶、星星或内容卡片，让星禾更像元气创造者。
- 太工具化：加入星禾的观察、判断或陪跑动作，让画面有个人 IP 温度。
- 文字错：优先局部编辑；错得多就重生成并减少标注数量。
- 失败返工：按 `failure-recovery-playbook.md` 输出失败类型、根因、修复动作和重写 prompt。

## 交付判断

高质量正文图、封面、情绪图和漫画应该让读者先记住“星禾正在做一件和内容有关的事”，然后 1 秒内看懂这个创作、运营或判断动作。

高质量技术架构图、流程图和知识卡片应该让读者先看懂结构、路径、节点或卡片重点；星禾只在不干扰阅读时作为小引导、局部动作或风格记忆点出现。

如果第一眼像模板课件、营销海报或冷硬架构图，而不是白纸上的星禾内容创作笔记、解释图、技术结构图、流程图或知识卡片，就不合格。
