# 全景信息图规则

本文件用于 `panoramic-infographic`：高密度、横向、一屏讲清系统全貌的信息图。它学习的是“构图和信息组织”，不是参考图的画风、品牌、图标、文字或配色。

## 适用场景

适合：

- 系统全景：输入、核心循环、能力支撑、输出、外部工具、风险和对比都需要同时出现。
- 方法论地图：一个概念需要解释“为什么重要、怎么运转、和旧方式有什么差异”。
- 技术/Agent 架构总览：既有架构层，也有执行循环、输出交付和工具生态。
- 产品能力总览：上层价值、核心模块、用户输入、系统输出、工程提示需要放在同一张图里。
- 文章内嵌横图：读者在桌面端或公众号正文中横向阅读，目标是先看全局，再读细节。

不适合：

- 只讲一个观点或一个情绪，用情绪图、解释图或单张知识卡更清楚。
- 需要手机竖向连续阅读，用知识卡片组拆分。
- 节点关系需要工程精确审计，用 Mermaid、Excalidraw 或技术架构图拆图。
- 文字超过 10 个信息区或 28 个短标签，先做全景图概览，再拆知识卡片组。

## 核心结构

全景信息图不是把全文塞进一张图，而是建立一条可扫描的“全局阅读路径”。

推荐结构：

1. **顶部价值带**：一句话主标题 + 一个价值公式 / 核心隐喻 / 关键判断。
2. **左侧输入栈**：用户输入、事件、文件、历史记录、外部数据等入口。
3. **中央主链路**：核心循环、核心架构、关键路径或主要方法，必须是画面最清楚的部分。
4. **下方支撑层**：能力模块、记忆、检索、权限、安全、调试、观察、工具箱等底座能力。
5. **右侧输出栈**：最终答案、执行动作、交付物、触发后续流程等结果。
6. **右侧解释栏**：一句话理解、为什么重要、对比表、工程提示或风险清单。
7. **底部工具带**：可选，用于列出工具类型、接口、资源、外部系统，不写具体密钥、域名或真实 logo。

典型阅读路径：

```text
顶部价值判断
-> 左侧输入
-> 中央核心循环 / 架构
-> 右侧输出
-> 右侧解释与对比
-> 底部工具箱或工程提示
```

## 构图骨架

按内容选择一种骨架，不要每次都画成同一种全景图。

### A. 中央核心循环型

适合 Agent Loop、自动化系统、内容生产闭环、运营闭环。

```text
顶部：价值公式 / 一句话解释
左：输入层
中：5 步核心循环
下：运行支撑层
右：输出层
右侧栏：无/有该机制的对比 + 工程 Tips
底部：工具箱
```

### B. 分层架构全景型

适合技术架构、平台能力、工作区系统、产品能力地图。

```text
顶部：系统目标
中上：接入层 / 协议层
中央：编排层 / 核心能力层
中下：存储层 / 记忆层 / 安全层
右侧：交付物 / 观测 / 运营闭环
底部：工具和外部系统
```

### C. 左旧右新对比全景型

适合“旧方式 vs 新方式”“没有系统 vs 有系统”“只用模型 vs Agent 工作流”。

```text
顶部：一句话结论
左半：旧方式的问题链
中间：转折机制 / 核心能力
右半：新方式的运行链
底部：落地条件和工具清单
```

### D. 地图导航型

适合学习路线、软件交付路径、内容自动化路径、项目阶段总览。

```text
顶部：路径名称
中央：从起点到终点的路线
上下：阶段目标 / 关键能力 / 产物 / 风险
右侧：读者如何使用这张图
底部：检查清单
```

## 信息密度

全景信息图允许比普通信息图更高密度，但必须可扫描。

- 默认信息密度：`high` 或 `panoramic-high`。
- 建议信息区：7-10 个。
- 建议短标签：18-28 个。
- 每个节点标题：2-8 个中文字符。
- 每个解释句：最多 1 行短句，不放段落。
- 中央主链路最多 5-7 个关键节点。
- 右侧解释栏最多 3 个小模块。
- 底部工具带最多 8 个工具类别。

超过上限时：

- 保留一张全景图，只讲“关系和地图”。
- 把细节拆成 `knowledge-card-pack`。
- 技术细节拆成 `technical-architecture` 或 `process-flow`。

## 人物呈现

全景信息图里信息结构是主体。

- 默认 `no-character` 或 `small-character`。
- 需要人物时，星禾只能作为小型导航员、标注者或边界检查者。
- 可放在右上角、底部工具带旁、中央主链路边缘或一个关键检查点旁。
- 不能让完整人物占据中心，也不能遮挡节点、箭头、公式、对比表或模块标题。
- 如果人物出现，会使用人物基准图；如果人物不出现，仍保留星禾风格的冷白/浅蓝灰手绘结构、批注和文字层级。

## 视觉层级

- 背景：冷白或极浅蓝灰。
- 模块底：浅蓝灰、浅青灰、低饱和浅色块，用来区分区域。
- 主链路：可以使用橙色文字、橙色箭头或橙色节点标题。
- 次级信息：深蓝、蓝灰或黑色。
- 风险、错误、缺失：少量红色。
- 不复制参考图的具体图标、车辆、盾牌、品牌、标题和文字。
- 不把所有文字都变成橙色；橙色可以直接作为标题字、关键词或主路径文字。

## 候选方向字段

使用 `panoramic-infographic` 时，每个候选方向必须写：

```json
{
  "primary_route": "panoramic-infographic",
  "secondary_routes": ["knowledge-card-pack"],
  "information_density": "panoramic-high",
  "text_density_level": "panoramic-high",
  "panorama_structure": "中央核心循环型",
  "section_count": 8,
  "label_budget": "8 个信息区 + 22 个短标签",
  "reading_path": "顶部价值 -> 左侧输入 -> 中央循环 -> 右侧输出 -> 侧栏对比 -> 底部工具",
  "character_presence": "small-character",
  "xinghe_action": "星禾在中央主链路边缘贴上检查标签，不遮挡节点",
  "aspect_ratio": "16:9",
  "overflow_plan": "细节拆成 5 张知识卡片"
}
```

## Prompt 必写项

全景信息图 prompt 必须包含：

- 主题和读者看完要理解的一句话。
- 顶部价值带写什么。
- 左侧输入栈有哪些。
- 中央主链路有哪些节点。
- 下方支撑层有哪些能力。
- 右侧输出栈有哪些。
- 右侧解释栏包含哪些短模块。
- 底部工具带是否需要。
- 人物是否出现、出现多小、做什么动作。
- 文字预算和拆分方案。

## 失败判断

看到以下问题要返工：

- 只是一张横向 PPT，缺少阅读路径。
- 文字像段落，无法快速扫描。
- 中央主链路不突出，所有区域平均用力。
- 右侧栏堆了太多解释，压过主图。
- 人物过大，抢走结构主体。
- 为了信息密度牺牲中文可读性。
- 复制了参考图的具体标题、图标、车辆、盾牌或模块内容。
