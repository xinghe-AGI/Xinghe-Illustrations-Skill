# 技术架构图与流程图规则

本文件用于技术架构图、流程图、节点关系图、SOP 流程和自动化流程。目标是让读者先看懂结构，再感受到星禾风格；不要为了展示人物牺牲信息清晰度。

## 路由

| 路由 | 适合内容 | 默认比例 | 人物呈现 |
|---|---|---|---|
| `technical-architecture` | 系统组件、服务边界、数据流、模块依赖、部署关系、权限边界 | `4:3` / `16:9` | `small-character` / `partial-character` / `no-character` |
| `process-flow` | SOP、自动化流程、审批流、状态流、内容生产流程、任务流 | `4:3` / `16:9` / 拆成 `3:4` 卡片组 | `small-character` / `partial-character` / `no-character` |
| `explanatory-diagram` | 轻量机制、3-5 步、概念关系 | `4:3` | `small-character` |
| `knowledge-card-single` | 单流程、单对比、单清单 | `4:3` / `16:9` / `3:4` / `1:1`，按发布场景和信息排版决定 | `small-character` / `partial-character` / `no-character` |

## 人物呈现等级

| 等级 | 用法 | 规则 |
|---|---|---|
| `full-character` | 封面、情绪图、正文锚点图 | 人物可占 25%-60%，必须承担动作 |
| `small-character` | 解释图、架构图、流程图、知识卡片 | 人物占 5%-15%，只做指向、标注、接线、拿笔、贴签 |
| `partial-character` | 高密度结构图、卡片、横向流程 | 只画手、半身、侧影或小头像旁的手写批注；不能抢信息层级 |
| `no-character` | 精确结构、节点较多、人物会干扰理解 | 不画人物，只保留冷白/浅蓝灰底蜡笔、手写标签、橙色主路径、蓝色反馈/数据线、红色风险标记 |

如果选择 `no-character`，不要把输出称为“星禾人物图”，应称为“星禾风格结构图”。如果画面中出现任何星禾人物、半身、手或头像，仍要使用人物基准图约束。

## 技术架构图

适合：

- 多个模块、服务、数据源、客户端、API、数据库、队列、模型、Agent、自动化任务之间的关系。
- 需要说明输入、处理、输出、存储、反馈、权限或部署边界。
- 读者需要理解“系统怎么连起来”，而不是只记住一个隐喻。

构图：

- 先画边界，再画节点：用户层、入口层、处理层、存储层、外部服务、反馈层。
- 节点建议 4-9 个；超过 9 个时拆成多张图或输出 Mermaid/Excalidraw 建议。
- 主数据流用橙色线；反馈、日志、状态回流用蓝色线；风险、权限、失败点用红色圈注；层级背景用极浅蓝灰块或手绘虚线分区，避免节点漂浮。
- 人物默认在角落或边缘，只做“小小的架构标注者”：拿笔圈边界、贴标签、指向关键路径。
- 如果人物会挤压节点文字，使用 `partial-character` 或 `no-character`。

禁忌：

- 不把技术架构图画成大人物 + 几个随意盒子。
- 不使用真实 App 截图、仿真 UI、品牌 logo、二维码、密钥、真实域名或真实 token。
- 不为了可爱感改变系统关系。
- 不把每条边都写长句；边上只放 1-4 字短标签。

## 流程图

适合：

- 从输入到输出的步骤、自动化链路、内容生产流程、审批/发布/复盘流程、任务状态流。
- 读者需要知道先后顺序、分支判断、回流点和失败点。

构图：

- 3-7 步优先做单张 `4:3` 或 `16:9`。
- 超过 7 步时，拆成知识卡片组或分段流程图。
- 主流程从左到右或从上到下，回流线用蓝色，关键断点用红色。
- 星禾可以只露一只手在关键节点贴“检查”“回流”“人工确认”等短标签。
- 流程节点用手绘圆角框、便签、小盒子，不做冷硬 BPMN/PPT 模板。

禁忌：

- 不把流程图做成课程页。
- 不把所有步骤都塞进同一张小红书卡片。
- 不让人物站在流程旁边摆 pose；人物要么小而有用，要么不出现。

## 知识卡片中的人物缩放

知识卡片的第一优先级是信息结构可读，不是默认竖版。小红书/微信竖图才按手机竖向阅读优先；文章笔记、方法论总结、流程/对比/结构卡可以优先横版。人物不是每张卡都要完整出现：

- 封面卡：可用 `full-character` 或 `small-character`。
- 总览卡、流程卡、对比卡：优先 `small-character` 或 `partial-character`。
- 高密度清单卡、术语卡、架构卡：允许 `no-character`。
- 一组卡片里不需要每张都出现完整星禾；可以 1-2 张完整，2-4 张小人物/局部，少数结构卡无人物。

## Prompt 必填字段

技术架构图、流程图和高密度知识卡片的候选方向必须写：

- `route`：`technical-architecture` / `process-flow` / `knowledge-card-single` / `knowledge-card-pack`
- `information_goal`：读者看完要理解什么
- `structure_type`：分层架构 / 数据流 / 状态流 / SOP / 自动化链路 / 输入-处理-输出
- `node_count`：预计节点数
- `edge_style`：主流程、反馈线、风险点如何区分
- `character_presence`：`full-character` / `small-character` / `partial-character` / `no-character`
- `why_this_presence`：为什么人物应该大、小、局部或不出现

## 验收标准

- 先看懂结构，再看见风格。
- 节点、方向、层级、反馈或边界清楚。
- 人物没有抢走主要信息。
- 人物出现时有明确动作；不出现时画面仍保持冷白/浅蓝灰底蜡笔、手写批注和星禾内容笔记气质。
- 复杂到无法读清时，主动拆图或建议 Mermaid/Excalidraw。
